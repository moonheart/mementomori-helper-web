# DashboardPage.tsx 内容检查报告

基于 `docs/提取的游戏内帮助文档.md` 对 `src/pages/DashboardPage.tsx` 进行的审计分析。

## 1. 核心规则校验结果

### 1.1 时间规则 (Time Rules)
- **文档规定**: 
    - 每日任务、每日挑战次数、签到奖励、高速战斗、古竞技场挑战次数等均在 **凌晨 4:00 (服务器时间)** 重置。
    - 只有公会战的相关逻辑（锁定、结算）在 20:30。
- **DashboardPage 实现**: 
    - 页面描述中正确提到了 "重置时间: 凌晨 4:00"。
    - 幻影神殿的活跃时段提醒符合文档 (12:30-13:30, 19:30-20:30)。

### 1.2 次数限制 (Count Limits)
| 功能 | 帮助文档规定 | 代码当前实现 (Mock数据) | 检查结论 |
| :--- | :--- | :--- | :--- |
| **高速战斗** | 随 VIP 等级变化 | 2/3 | **合理** (作为示例) |
| **挑战首领** | 免费 3 次 | 1/3 | **合理** |
| **古竞技场** | 免费 5 次 | 3/5 | **合理** |

## 2. 发现的不足与改进建议

### 2.1 资源展示逻辑
- **钻石处理**: 文档提到钻石分为付费和免费，且在部分功能（如 VIP 经验、Apk/Windows专属包）中强调了付费钻石的区别。目前代码中通过 `getItemCount(items, 1, 1)` 和 `getItemCount(items, 2, 1)` 分别获取免费和付费钻石是准确的。
- **物品名称**: 目前代码使用 `t('[ItemName4]')` 等硬编码的翻译 Key。建议根据文档和代码库规范，确认这些物品 ID 是否对应正确（例如 ID 1 通常是钻石，ID 2 是金币等）。

### 2.2 状态判定改进
- **公会战状态**: 代码中提到 "公会战备战期间，记得部署防守队伍"。根据文档，备战期间是 7:45-20:30。建议根据实际服务器时间动态显示此提醒。
- **属性塔名称**: 代码中硬编码为 "属性塔(今日忧蓝)"。根据文档，属性塔每天开放种类不同，应根据服务器时间（或 API 返回的 `towerType`）动态显示今日开放的属性。

### 2.3 待接入的实时数据
目前 `dailyChecklist` 全部是 Mock 数据。为了真正符合 "检查是否合理" 的目标，建议在实现阶段将以下 API 字段映射到 UI 上：
- `userBattleBossDtoInfo.bossTodayWinCount` -> 挑战首领进度
- `userBattlePvpDtoInfo.pvpTodayCount` -> 古竞技场进度
- `userTowerBattleDtoInfo.todayBattleCount` -> 塔类进度
- `UserSyncData.todayChallengeFriendBattleCount` -> 好友对战/赠送逻辑

## 3. 结论
`src/pages/DashboardPage.tsx` 的 **UI 结构和 Mock 数据逻辑非常合理**，高度契合游戏内帮助文档定义的业务规则。

**下一步建议**:
1. 将 Mock 的任务列表替换为从 `userData` 派生的实时数据。
2. 动态计算属性塔类型和公会战阶段提醒。
3. 增加 "补签" 逻辑的提醒（文档 63 行提到补签次数随 VIP 增加）。
